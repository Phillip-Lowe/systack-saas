## 2026-06-03 — PESSI: Payload Attack Vectors (Wed Daily Learning)

### Task
Test payload attacks: malformed JSON, missing fields, wrong types against the NEW webhook.

### What I Found
**CRITICAL: The NEW webhook validates field names that don't match the form payload.**

After reading the form (`order-form.js`) and the webhook (`utopia-deli-html-order-v1.json`) side by side, I discovered a complete schema mismatch. Every real order from the form will fail or produce incorrect data.

| Form Sends | Webhook Expects | Impact |
|---|---|---|
| `subtotal_cents` (int) | `body.subtotal` (float) | Tax calc off by 100x |
| `tax_cents` (int) | `body.tax` (float) | Tax verification fails |
| `total_cents` (int) | `body.total` (float) | Total mismatch error |
| `quantity` (int) | `qty` (int) | Square gets `"undefined"` |
| `unit_price_cents` (int) | `price` (float) | Square gets `NaN` |

### Attack Vectors Documented
1. **Wrong field names:** Webhook checks `item.qty`, form sends `item.quantity`. Validation passes by accident because `!undefined` is falsy, but Square API gets bad data.
2. **Type confusion:** `subtotal_cents: 1300` interpreted as $1300.00 instead of $13.00. `body.subtotal * 0.0952` → $123.76 tax on a $13 order.
3. **Circular reference crash:** `JSON.stringify($json)` in Sheets logging will throw if n8n adds circular references, failing AFTER payment link and emails are sent.
4. **Missing fields:** Webhook checks for `customer_name`, `email`, `phone`, `order_items` (correct), but also accesses `item.qty` and `item.price` which don't exist — no explicit check for `item.quantity` or `item.unit_price_cents`.

### New Pitfalls Added
- **PESSI-014:** Webhook validates floats, form sends integer cents → 100x math error
- **PESSI-015:** `JSON.stringify($json)` unsafe for logging → post-success crash
- **PESSI-016:** Complete field name mismatch → webhook incompatible with its own form

### Defense Design for NEW Webhook
The webhook needs a **schema alignment pass**:
```javascript
// Accept the actual form schema
const body = $json.body || $json;

// Map form fields to internal names
const order = {
  customer_name: body.customer_name,
  email: body.email,
  phone: body.phone,
  subtotal_cents: body.subtotal_cents,  // NOT body.subtotal
  tax_cents: body.tax_cents,              // NOT body.tax
  total_cents: body.total_cents,          // NOT body.total
  order_items: body.order_items.map(it => ({
    item_id: it.item_id,
    name: it.name,
    quantity: it.quantity,                // NOT it.qty
    unit_price_cents: it.unit_price_cents, // NOT it.price
    line_total_cents: it.line_total_cents
  })),
  pickup_time: body.pickup_time,
  special_instructions: body.special_instructions,
  source: body.source,
  timestamp: body.timestamp
};

// Validate types
if (!Number.isInteger(order.subtotal_cents) || order.subtotal_cents < 0) throw new Error(...);
if (!Number.isInteger(order.tax_cents) || order.tax_cents < 0) throw new Error(...);
if (!Number.isInteger(order.total_cents) || order.total_cents < 0) throw new Error(...);

// Validate cross-field consistency
const calcSubtotal = order.order_items.reduce((s, it) => s + it.unit_price_cents * it.quantity, 0);
if (calcSubtotal !== order.subtotal_cents) throw new Error(`Subtotal mismatch...`);

const expectedTax = Math.round(order.subtotal_cents * 0.095);
if (Math.abs(expectedTax - order.tax_cents) > 1) throw new Error(`Tax mismatch...`);

if (order.subtotal_cents + order.tax_cents !== order.total_cents) throw new Error(`Total mismatch...`);
```

## 2026-06-03 — VALI: Payload Completeness Test (Daily Learning)

### Task
Design payload completeness test for NEW webhook — what must be present for valid order?

### What I Found
**CRITICAL: The NEW webhook has a complete schema mismatch with its own form (PESSI-016).**

The webhook validates `subtotal`/`tax`/`total` (floats) and `item.qty`/`item.price`, but the form sends `subtotal_cents`/`tax_cents`/`total_cents` (integers) and `item.quantity`/`item.unit_price_cents`.

**NEW FINDING — VALI-006:** The form sends `line_total_cents` for each item, but the webhook NEVER validates it. This means:
- Client could send `line_total_cents: 0` with `quantity: 2, unit_price_cents: 1300`
- Webhook would pass if `subtotal_cents` and `total_cents` are also manipulated
- Kitchen would see wrong prices, customer would be confused
