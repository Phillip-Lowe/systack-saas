## Architecture Decisions Made (Need ORACLE Review)

### 1. RAM Sizing — Is 16GB Enough?

**Decision:** 16GB VPS for Business tier (qwen2.5:7b = 4.4GB model)

**Math:**
- Model: 4.4GB
- OS + Gateway: 1GB
- n8n + Docker: 0.8GB
- Total: ~6.2GB used
- Free: ~9.8GB headroom

**Question for ORACLE:** Should we offer 14B upgrade (24GB VPS +$96/mo) or keep 7B as standard?

### 2. Tailscale Multi-Client Architecture

**Decision:** Use **tagged devices** instead of user seats
- Free tier: 6 users, unlimited tagged devices
- Client VPS = tagged device (no user account needed)
- Client access: HTTPS URL via Tailscale Serve/Funnel

**ORACLE question:** Is this sustainable at 100 clients? 500?

### 3. Agent Runs on VPS, Not Client Computer

**Decision:** Cloud-native deployment
- Agent on Vultr VPS (16GB)
- Client accesses via Tailscale URL
- Agent can email/API scrape/process files but NOT access local PC

**ORACLE question:** Do we need an "on-premise" option for clients who want agent on their local network?

### 4. n8n Workflow Architecture

**Current flow:**
```
Stripe webhook → n8n → Postgres saos_clients → task_queue → 
ASSEMBLY agent → Vultr API → VPS creation → cloud-init → 
Tailscale join → Ollama install → n8n install → webhook callback → 
VALI health check → CHATTY email send
```

**ORACLE question:** Is this the right sequence? Should health check happen BEFORE or AFTER template deployment?
